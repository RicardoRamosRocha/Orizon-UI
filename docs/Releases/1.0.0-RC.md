# Release Candidate 1.0.0

Data da validação: 2026-07-21

## Resumo executivo

A Orizon UI está apta para publicação controlada como `1.0.0-rc.1`. O pacote principal, o pacote de símbolos, os Static Web Assets, os Tag Helpers e a integração em um projeto MVC novo foram validados sem dependência do Sandbox. Não houve alteração intencional de API pública, nomes de Tag Helpers, tokens documentados ou comportamento de componentes.

A promoção para a versão estável `1.0.0` mantém dois gates explícitos: concluir uma passagem visual manual da matriz de viewports em um ambiente de browser disponível e decidir a política de cobertura XML da superfície pública histórica. O build do RC produz o XML e mantém `CS1591` suprimido para membros legados ainda sem comentário.

## Build

- `dotnet restore Orizon.UI.sln`: aprovado.
- `dotnet build Orizon.UI.sln --configuration Release`: aprovado, 0 erros e 0 warnings.
- `dotnet pack`: aprovado para `Orizon.UI.1.0.0-rc.1.nupkg` e `Orizon.UI.1.0.0-rc.1.snupkg`.
- Build determinístico, SourceLink, repositório, README, ícone, licença MIT e documentação XML confirmados no pacote.

## Testes

- `dotnet test Orizon.UI.sln --configuration Release --no-build`: 12 aprovados, 0 falhas.
- Cobertura existente validada: sorting, filtering, grouping, paging, selection, clipboard, undo, redo, virtualization, batch, server mode e exportação XLSX do Enterprise Grid.
- Sintaxe JavaScript: 29 arquivos verificados, 0 falhas.
- Validador ODL: aprovado.

## API Review

- Assinaturas públicas e nomes de Tag Helpers preservados.
- Contratos centrais do Enterprise Grid, provider em memória e registro de ícones receberam documentação XML.
- Defaults e parâmetros não foram modificados.
- A geração do arquivo XML foi habilitada no projeto e o arquivo está presente em `lib/net8.0` no pacote.
- Dívida conhecida: a superfície histórica ainda contém membros públicos sem XML; `CS1591` permanece explicitamente suprimido para evitar alterações mecânicas de alto risco durante o feature freeze.

## Design Tokens

- 704 declarações e 647 nomes únicos revisados em `design-tokens.css`.
- As 57 repetições encontradas são overrides escopados de densidade, radius, motion e alto contraste, não duplicações acidentais.
- 161 tokens aparecem apenas na declaração; foram preservados por funcionarem como aliases públicos ou pontos de extensão de tema.
- Light, dark e alto contraste permanecem cobertos sem remoção de contrato visual.

## CSS

- 53 arquivos, 277.509 bytes de fonte CSS.
- 22 usos de `!important` revisados; concentram-se em estados hidden, reduced motion, gráficos, impressão e regras responsivas.
- 0 seletores por ID encontrados.
- 47 imports no entry point e 64 imports no conjunto de bundles foram mantidos para não alterar a arquitetura no RC.
- Nenhuma consolidação de risco foi aplicada e a aparência não foi intencionalmente alterada.

## JavaScript

- 29 arquivos, 95.046 bytes de fonte JavaScript.
- Entry point modular com 17 imports revisado.
- Inicialização, delegação de eventos e cleanup foram inspecionados; não foi identificada correção segura que justificasse mudança de runtime.
- Nenhuma função pública ou comportamento foi alterado.

## Acessibilidade

- Revisados `aria-label`, `aria-expanded`, `aria-controls`, `aria-live`, roles, foco visível, ordem de tabulação e tratamento de teclado.
- Padrões de Escape, Enter, Space e setas estão presentes nos componentes interativos aplicáveis.
- Regras globais e locais para `prefers-reduced-motion` foram confirmadas.
- Estados light/dark e alto contraste foram inspecionados por fonte; não houve alteração visual no RC.

## Responsividade

- Breakpoints globais e específicos cobrem desktop, notebook, tablet, mobile e landscape.
- Sidebar, topbar, grid, cards, forms, Studio, Patterns e Templates possuem regras responsivas.
- Renderização HTTP e assets foram validados no consumidor. A automação visual por viewport não pôde ser executada porque o browser integrado do ambiente não iniciou devido a uma restrição ACL; permanece necessária uma passagem visual manual antes da promoção estável.

## Performance

- Tamanho do `.nupkg`: 217.225 bytes.
- Tamanho do `.snupkg`: 50.680 bytes.
- Consumidor local, 10 requests: média de 71,14 ms, mínimo de 40,81 ms e máximo de 278,26 ms, incluindo aquecimento.
- Consumidor local, 20 requests aquecidos: mediana de 47,83 ms e p95 de 53,29 ms.
- Backlog conhecido: 47 imports CSS e 17 imports JS no carregamento modular. Bundling não foi introduzido por caracterizar mudança arquitetural fora do escopo do RC.

## Projeto consumidor

Um projeto ASP.NET Core MVC novo foi criado com `dotnet new mvc --framework net8.0`, recebeu somente o pacote local `Orizon.UI 1.0.0-rc.1` e foi compilado em Release com 0 erros e 0 warnings.

Validações aprovadas:

- Static Web Assets CSS e JavaScript responderam HTTP 200 em `/_content/Orizon.UI`.
- Registro `@addTagHelper *, Orizon.UI` funcionou a partir do pacote.
- `_AdminLayout`, sidebar, topbar, button, card, forms e Enterprise Grid foram renderizados.
- Tema light foi aplicado sem copiar arquivos do Sandbox.

O projeto temporário foi removido após a validação; o script `scripts/validate-local-package.ps1` reproduz o fluxo.

## Pendências para 1.0.0 estável

1. Executar matriz visual manual em desktop, notebook, tablet, mobile e landscape quando um browser de automação estiver disponível.
2. Definir e executar a meta de cobertura XML dos membros públicos históricos antes de remover a supressão `CS1591`.
3. Tratar bundling/minificação somente em sprint posterior, com baseline visual e de performance, pois isso altera a estratégia de entrega dos assets.
